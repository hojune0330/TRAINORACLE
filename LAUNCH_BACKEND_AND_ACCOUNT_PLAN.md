## 1. 왜 지금 백엔드가 "없어도" 서비스를 시작할 수 있는가

핵심 판단: **저장을 두 단계로 나눈다.**

### 1단계 — 기기 로컬 저장 (회원가입 불필요, 토큰 불필요, 즉시 가능)

- 훈련일지·체크인 입력을 **사용자 기기 안에**(브라우저 localStorage/IndexedDB) 저장.
- 회원가입 없이 앱 열자마자 바로 쓰기 시작 → 초기 사용자 진입장벽 0.
- 개인정보가 서버에 없으므로 **개인정보처리방침·보안 부담이 최소**인 상태로 실사용 검증 가능.
- 한계: 기기 바꾸면 데이터 이동 안 됨, 코치가 원격으로 못 봄. (→ 2단계에서 해결)
- PWA(홈 화면 설치)와 결합하면 사실상 "앱처럼" 동작.

### 2단계 — 계정 + 서버 동기화 (간편가입, 토큰 필요)

- 로컬에 쌓인 데이터를 계정 생성 시 서버로 **승격(업로드)** — "가입하면 지금까지 쓴 일지가 그대로 계정에 붙는" 흐름.
- 코치-선수 공유, 다기기 동기화, 분석 탭의 실데이터가 이 단계에서 열림.

이 순서면 **회원 시스템이 완성되기 전에도 서비스가 돌아가고**, 회원 시스템은 뒤에서 조용히 준비할 수 있다.

## 2. 백엔드 아키텍처 결정 (권장안)

### 권장: Cloudflare (Pages + Workers + D1)

| 후보 | 장점 | 단점 | 판정 |
|---|---|---|---|
| **Cloudflare Workers + D1** | 무료 티어 넉넉, 프론트와 같은 곳에 배포, `wrangler.jsonc` 이미 준비됨, SQL DB(D1) 포함 | 사용자 API 토큰 1회 필요 | ✅ **채택** |
| Supabase | 인증(소셜 로그인) 내장, Postgres | 계정·프로젝트 별도 생성 필요, 무료 티어 절전(pause) 이슈 | 보류(인증만 차용 검토 가능) |
| Firebase | 인증·푸시 강력 | 구글 종속, 데이터 모델(NoSQL)이 우리 계약서와 안 맞음 | 탈락 |
| 자체 서버(VPS) | 자유도 최대 | 운영 부담·비용, 1인 운영에 과함 | 탈락 |

구성:

```
[프론트: Cloudflare Pages]  ←→  [API: Cloudflare Workers]  ←→  [DB: D1 (SQLite)]
     (현 GitHub Pages에서 이전)          /api/*                    journal, checkin,
                                                                  account, consent_audit
```

- GitHub Pages는 **정적 파일만** 되므로, 백엔드가 붙는 시점에 Cloudflare로 이전한다. 이전 전까지 GitHub Pages 유지(주소 공지 최소화 권장).
- 안전 불변식(D9 차단 등)은 계약서대로 **서버 응답이 안전 판정을 바꿀 수 없음** — 백엔드는 저장/동기화만 담당.

## 3. 회원가입 설계 — 간편가입

### 3.1 원칙

- **이메일+비밀번호 가입은 만들지 않는다** (관리 부담·피싱 위험·사용자 귀찮음).
- 소셜 간편가입만 제공. 한국 육상부 학생·코치 사용자층 기준 우선순위:

| 순위 | 제공자 | 이유 | 필요한 것 (사용자 액션) |
|---|---|---|---|
| 1 | **카카오** | 국내 10~20대 보급률 최고 | Kakao Developers 앱 생성 → REST API 키 + Redirect URI 등록 |
| 2 | **Apple** | iOS 앱스토어 출시 시 **의무**(소셜 로그인 제공 시) | Apple Developer 계정($99/년) — 스토어 출시 단계에서 어차피 필요 |
| 3 | Google | 안드로이드 커버 | Google Cloud Console OAuth 클라이언트 생성 |
| — | 네이버 | 선택 | 후순위, 초기 미제공 |

### 3.2 미성년자 이슈 (중요)

사용자층에 중·고교 선수가 포함될 가능성이 높음 → **만 14세 미만은 법정대리인 동의 필요**(개인정보보호법).
- 초기 대응: 가입 시 생년 확인 → 14세 미만이면 가입 보류 + 보호자 동의 절차 안내.
- 이는 EXTERNAL_RECORD_INTEGRATION_SPEC의 open issue **OI-ERI-GUARDIAN-CONSENT-001**과 같은 계열 — 계정 시스템 설계 시 함께 해소한다.

### 3.3 계정 데이터 최소주의

- 저장: 소셜 제공자 ID(해시), 표시 이름, 생년(연령 확인용), 소속(선택).
- 저장 안 함: 전화번호, 주소, 실명 강제 없음.

## 4. 애슬리트타임 회원 연동 — 검토 결과

### 4.1 사실 확인 (2026-07-10 조사)

- 공개 웹 검색: 애슬리트타임의 **개발자용 API·OAuth·제휴 문서 발견되지 않음**.
- 추정 도메인 프로브(athletetime.co.kr / .kr / .net 등): 모두 미응답 — 정확한 서비스 도메인 자체가 미확인 상태.
- 결론: **"애슬리트타임 계정으로 로그인"(OAuth SSO)은 현재 기술적으로 불가능** — 그쪽이 인증 서버를 열어줘야만 가능한 방식이기 때문.

### 4.2 그래서 가능한 3가지 경로 (단계적)

| 경로 | 내용 | 가능 시점 | 필요한 것 |
|---|---|---|---|
| **A. 수동 기록 입력** | 선수가 자기 PB/SB를 직접 입력, "출처: 애슬리트타임(수동)" 라벨 | **즉시** | 없음 |
| **B. 조회 링크/화면 확인** | 앱에서 애슬리트타임 기록 페이지로 링크 → 선수·코치가 보고 옮겨 적음 | 즉시(정확한 URL 확인 후) | 사용자: 애슬리트타임 정확한 서비스명/주소 확인 |
| **C. 공식 제휴 API 연동** | PB/SB 자동 조회 (계약서 EXTERNAL_RECORD_INTEGRATION_SPEC 준수) | 제휴 성사 후 | **사용자: 애슬리트타임 운영사 접촉·제휴 협의** (개발자가 아닌 사업 액션) |

- C가 성사되어도 계약서 원칙 유지: **일방향 인바운드, PB/SB만, 안전 판정 권한 없음(`can_affect_safety_disposition: false`), 동의 감사 기록**.
- "애슬리트타임 회원과 연동"의 현실적 의미: 당장은 **우리 계정에 애슬리트타임 기록을 붙이는 것(A→B→C)**이지, 그쪽 계정으로 로그인하는 게 아님.

### 4.3 사용자 확인 요청 (블로킹 질문)

> **Q1.** "애슬리트타임"의 정확한 서비스 주소(URL) 또는 앱 이름을 알려주세요. (검색·도메인 조사로 특정 실패)
> **Q2.** 애슬리트타임 운영사와 접촉 가능한 관계가 있나요? (제휴 API는 그쪽 협조 없이는 불가)

## 5. 실행 로드맵 (승인 시 즉시 착수 순서)

| 단계 | 작업 | 토큰/사용자 액션 | 담당 |
|---|---|---|---|
| L1 | **PWA** — manifest + service worker, 홈 화면 설치 | 없음 | 총책임자 (즉시) |
| L2 | **로컬 저장** — 일지/체크인 localStorage 저장·불러오기 | 없음 | 총책임자 (즉시) |
| L3 | **Cloudflare 이전** — Pages+Workers 배포 | 🔑 사용자: Cloudflare API 토큰 1회 (아래 6.1) | 총책임자 |
| L4 | **간편가입(카카오)** — OAuth 연동 | 🔑 사용자: 카카오 개발자 앱 키 (아래 6.2) | 총책임자 |
| L5 | **서버 동기화** — D1 스키마 + 로컬→서버 승격 | L3 완료 후 자동 | 총책임자 |
| L6 | **애슬리트타임 A/B 경로** — 수동 입력 + 링크 | 사용자: Q1 답변 | 총책임자 |
| L7 | **스토어 패키징** | 🔑 사용자: Apple($99/년)·Google($25) 개발자 계정 | 총책임자 + 사용자 |
| L8 | **애슬리트타임 C 경로(제휴 API)** | 사용자: 운영사 제휴 협의 | 사용자 주도 |

L1, L2는 **어떤 토큰도 필요 없어 지금 바로** 진행 가능. L3부터 사용자 도움이 필요.

## 6. 🔑 사용자 액션 체크리스트 (간편하지만 토큰이 필요한 부분)

### 6.1 Cloudflare API 토큰 (L3 — 백엔드 배포용, 5분)

1. https://dash.cloudflare.com 가입(무료) 또는 로그인
2. 우측 상단 프로필 → **My Profile → API Tokens → Create Token**
3. 템플릿 **"Edit Cloudflare Workers"** 선택 → Continue → Create Token
4. 발급된 토큰을 이 프로젝트의 **Deploy 패널**에 입력 (채팅에 직접 붙여넣지 말 것)
- 비용: 무료 티어로 충분 (Workers 10만 요청/일, D1 500만 읽기/일)

### 6.2 카카오 개발자 앱 (L4 — 간편가입용, 10분)

1. https://developers.kakao.com 로그인 → **내 애플리케이션 → 애플리케이션 추가**
2. 앱 이름 "TRAINORACLE" → 생성
3. **앱 키** 중 `REST API 키`를 복사해 전달 (비밀키 아님, Redirect 방식이라 노출 위험 낮음 — 그래도 Deploy 패널/비공개 경로 권장)
4. **카카오 로그인 활성화** → Redirect URI에 서비스 주소 등록 (주소는 L3 완료 후 내가 알려드림)
5. 동의항목: 닉네임만 필수로 설정 (최소 수집)
- 비용: 무료

### 6.3 스토어 계정 (L7 — 나중에)

- Apple Developer Program: $99/년 (iOS 출시 + Apple 로그인)
- Google Play Console: $25 (1회)
- 심사 제출 시 개인정보처리방침 URL 필요 → L4 전에 내가 초안 작성

### 6.4 애슬리트타임 (L6/L8)

- Q1: 정확한 서비스 주소/앱명 회신
- Q2(선택): 운영사 제휴 접촉 가능 여부

## 7. 이 문서의 지위

- 본 문서는 **결정 계획서**다. L1·L2는 별도 승인 없이 착수한다(토큰·비용·외부 의존 없음).
- L3 이후는 각 토큰/액션이 준비되는 즉시 순서대로 착수하며, 매 단계 커밋→PR→링크 공유 원칙을 유지한다.
- 안전 불변식·계약서 원칙(§8 memo_policy, is_safety_authority:false, 기준:데모 배지)은 모든 단계에서 불변.
